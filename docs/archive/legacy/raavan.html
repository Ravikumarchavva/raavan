✅ Normal Execution Flow — 11 steps
1
Browser
User types message → POST /api/chat with threadId & message
2
Next.js BFF
Proxies request, opens persistent SSE pipe back to browser
3
FastAPI
Registers SSE channel keyed by conversationId, calls RestateWorkflowClient
4
Restate
Durably accepts workflow start, journals it, dispatches to Worker
5
Worker
restore_memory() → loads full conversation history from Redis
6
OpenAI
do_llm_call() → AssistantMessage returned, result journalled by Restate
7
Worker
do_tool_exec() → tool runs, result journalled; loop repeats until end_turn
8
Redis + PG
persist_message() → message written to both stores atomically
9
NATS
publish_event() → text_delta / tool_result event fanned out to all subscribers
10
SSE Bridge
WebHITLBridge receives NATS event, pushes to matching SSE channel
11
Browser
EventSource fires, token/tool bubble rendered in real time ✨
⏸️ HITL Approval Flow — 7 steps
1
Worker
Agent selects a tool with requires_approval=True (e.g. send_email)
2
Restate
ctx.promise("hitl-{id}") created — Worker thread suspends, freed
3
NATS → Bridge
tool_approval_request event published and pushed over SSE
4
Browser
ToolApprovalCard rendered with tool name, inputs, and Approve / Reject buttons
5
Browser
User clicks Approve → POST /api/chat/respond/{requestId}
6
FastAPI → Restate
resolve_promise("hitl-{id}", approved:true) — Restate wakes the workflow instantly
7
Worker
ctx.promise().get() returns approved value — tool executes from exact pause point
Same pattern for ask_human tool
Instead of ToolApprovalCard → HumanInputCard shown. User types answer. Same promise resolution path. Worker receives text answer as ToolExecutionResultMessage.
🛡️
Why Restate? — Crash-Safe by Construction
If the Worker process dies between any two activity steps, Restate replays the durable journal and resumes from the last committed result. The LLM is never called twice. Tool side-effects are idempotency-keyed. HITL promises survive restarts. Your conversation state is never lost — even during deploys.
